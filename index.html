This report asks one simple question: which remote-signing risks still exist even when current ETSI controls are in place?

Core point: a signature can pass technical checks, but there can still be a dispute about whether the signer approved that exact content.

This document does not solve those risks. It shows where they remain and what evidence is missing when disputes happen.

Scope and Method

Scope: qualified remote signatures where signing is triggered through a service run by a trust service provider under eIDAS [[EIDAS910]] [[EIDAS2024_1183]].

Threat qualification rule:

A scenario is included only when it can still happen even if current ETSI controls are in place.

Attacker classes in scope:

  • A compromised or impersonating app/frontend with a valid session.
  • Privileged insider misuse in internal remote-signing systems.
  • Missing record of exact trust data used at validation time.
  • Activation evidence that stays internal and cannot be independently replayed by third parties.

Method:

Each scenario is presented in one consistent format: THREAT, MITIGATION, and RESIDUAL.

The goal is simple: show which controls are already in force, what an independent party can verify today, and what still remains disputed.

Terminology and Abbreviations

eIDAS
EU regulation for electronic identification and trust services (Regulation (EU) No 910/2014 and amendments) [[EIDAS910]] [[EIDAS2024_1183]].
TSP / QTSP
Trust Service Provider. A QTSP is a qualified trust service provider under eIDAS.
QSCD
Qualified Signature Creation Device. The protected signing device or service boundary used to create qualified signatures under eIDAS [[EIDAS910]].
ETSI / CEN
European standards bodies. ETSI publishes the remote-signing technical standards referenced here; CEN standards define related server-signing concepts.
DTBS
Data To Be Signed. In plain terms: the exact content that should be signed.
DTBSR
Data To Be Signed Representation. In plain terms: the representation of what is to be signed that remote-signing components process [[ETSI119432]] [[ETSI119431_1]].
SAD
Signature Activation Data. The activation material used to authorize a remote signing operation [[ETSI119431_1]].
SAM / SAP
In server-signing architecture, SAM is the server-side gate that decides whether a signing activation is allowed. SAP is the activation path (rules/messages) that carries that authorization context [[CENETSI2024]] [[EN4192412PP]].
OCSP / CRL
Certificate status sources used in validation: Online Certificate Status Protocol (OCSP) and Certificate Revocation List (CRL).
TL / LOTL
Trusted List (TL) and List of Trusted Lists (LOTL) used to discover trusted service status in the EU framework [[TL2015_1505]].
Sole control
The requirement that only the signer can validly trigger signature creation for that event, under the stated assumptions [[EIDAS910]].
Intent
Evidence that the signer approved the same content that was actually signed.
Verifier-checkable
Evidence a third party can independently replay and verify without relying only on provider narrative.
Assurance-only
A control mainly supported by audits, policy, or process evidence rather than standalone replayable artifacts.

Legal and Standards Baseline

eIDAS Article 25 says an electronic signature cannot be denied legal effect only because it is electronic [[EIDAS910]].

eIDAS Article 26(c) and Annex II require signature-creation data to be controlled and protected so others cannot validly trigger signing events [[EIDAS910]].

Regulation (EU) 2024/1183 updates eIDAS and sets the legal framework for remote qualified-signature components and related transition rules [[EIDAS910]] [[EIDAS2024_1183]].

Implementing Regulation (EU) 2025/1567 points to ETSI standards as the technical baseline used in this report [[EIDAS2025_1567]] [[ETSI119431_1]] [[ETSI319401]].

When EU law pins a specific standard version, that pinned version is the legal reference for conformance checks [[EIDAS2025_1567]].

ETSI TS 119 432 defines the remote-signing API model, but it does not itself define the full client authentication method [[ETSI119432]].

These controls reduce risk, but they do not automatically make every signer-intent claim independently replayable in a dispute.

System Model and Trust Boundaries

Assets:

  • Signature key activation state.
  • SAD and activation decisions.
  • DTBS/DTBSR linkage from display to signature input.
  • Validation-time trust material (OCSP/CRL/TL/LOTL).
  • Audit and event logs.

Main boundary split:

  • What third parties can usually verify directly: signature bytes, certificate chain, revocation status, and trusted-list state [[EIDAS910]] [[TL2015_1505]].
  • What often stays inside the provider: user-intent path details, activation internals, and internal log semantics [[ETSI119431_1]] [[ETSI319401]] [[ETSI119431_2]].

This split is the main reason residual dispute risk remains.

Residual Scenarios and Evidence Gaps

This report keeps four residual scenarios that can still exist under the current ETSI baseline.

Format per scenario: THREAT, MITIGATION (controls in force), and RESIDUAL (what still cannot be independently proven).

R1: Show A, Sign B

THREAT: A compromised frontend keeps valid authentication/session and submits content B after showing content A.

MITIGATION: Session and activation controls plus remote-signing API model [[ETSI119431_1]] [[ETSI119432]].

RESIDUAL: Independent proof that signer saw and approved the same document and terms that were finally signed.

R2: Privileged Internal Misuse

THREAT: Someone with high internal access abuses that access and gets a valid signature created, even though no key is exported.

MITIGATION: ETSI requires access control, role separation, monitoring, and audit controls for service components [[ETSI319401]] [[ETSI119431_2]].

RESIDUAL: In a dispute, an external reviewer may still lack event-level proof that this exact activation was free from privileged misuse.

R3: Missing Validation Snapshot

THREAT: At decision time, the service does not keep a complete record of the trust data it used (for example certificate status and trusted-list state).

MITIGATION: Validation and trusted-list framework [[EIDAS910]] [[TL2015_1505]].

RESIDUAL: Later replay may not match the original decision because the exact validation inputs are not fully reproducible.

R4: Evidence Stays Inside the Provider

THREAT: Critical activation evidence remains internal, so third-party dispute replay is incomplete.

MITIGATION: Logging and governance obligations [[ETSI319401]] [[ETSI119431_2]].

RESIDUAL: External parties may still be unable to verify the full intent-related event history without relying on provider narrative.
